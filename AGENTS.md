# AGENTS.md

## Project Identity

本项目是“桃花潭景区数字化展示与预约平台”，目标是做成一个可展示、可答辩、可部署的高端文旅数字平台前端项目。
本项目定位为“安徽泾县桃花潭、查济、太平湖文旅数字化展示与预约平台”，重点体现皖南山水、徽派建筑、江南水乡与诗意文旅气质。

## Core Modification Rules

1. 技术栈保持 `Vite + React + TypeScript + React Router + Tailwind CSS`。
2. 不要迁移到 `Next.js`，也不要引入会改变项目运行形态的重型框架。
3. 不要从零重写项目，应基于现有代码逐步改造和增强。
4. 不要删除已有页面和模块；如需调整，优先做保留式优化、补全和修复。
5. 不要大规模改变现有信息架构；导航、页面分组、模块层级应尽量延续当前结构。
6. 每次修改后必须运行构建检查，默认执行 `npm run build`（在当前终端环境可等价使用 `npm.cmd run build`）。
7. 修改优先级始终为：
   - 能运行
   - 能构建
   - 能部署
   - 刷新子页面不 `404`
   - 站内跳转不穿帮
8. 视觉风格必须保持“中国高端文旅”方向，重点体现“徽派建筑、江南水乡、桃花潭、查济、太平湖”的整体气质。
9. 登录、注册、支付、订单、留言、发布等流程，现阶段只作为前端 mock 演示闭环处理，不接真实后端，不接真实支付，不伪造真实生产能力。
10. 每轮修改结束后，必须明确说明：
    - 改了哪些文件
    - 为什么这样改

## Implementation Guidance

- 优先修复会影响演示、答辩和部署稳定性的关键问题，而不是先做表面美化。
- 优先复用现有页面、组件、路由和样式资源，不轻易推翻重做。
- 若需要补充假数据、演示态状态或前端模拟流程，应保持内容自洽、跳转闭环完整、展示可信。
- 若某项改动可能影响现有页面结构、导航逻辑或答辩展示路径，应先遵守“最小必要改动”原则。

## Working Agreement For Future Rounds

后续所有改造默认受本文件约束。除非用户明确要求调整方向，否则应把本文件作为项目级工作规则执行。
